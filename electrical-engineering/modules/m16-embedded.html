Microcontrollers & Embedded Systems · Eingebettete Systeme

Over 99 % of all processors manufactured never sit in anything you would call a computer. They hide in washing machines, insulin pumps, e-bike controllers and the ~100 control units of a modern car — sensing, deciding and actuating on millisecond deadlines for years without a reboot. This module teaches you how those invisible computers work, from the fetch–decode–execute cycle to interrupts, PWM, serial buses and battery budgets.

Corresponds to: Mikrocomputertechnik / Eingebettete Systeme (4th–5th semester) Prerequisites: Module 10 (digital logic, flip-flops, state machines) Study time: ≈ 6–8 h

1. The invisible computer

An embedded system (eingebettetes System) is a computer built into a product to do one job — as opposed to a general-purpose PC that runs whatever you install. The job defines everything: a pacemaker prioritizes reliability and microamp power draw; an engine controller prioritizes guaranteed 100 µs reaction times; a smart-lamp controller prioritizes a 30-cent unit price. The engineering discipline is choosing just enough computer.

Some orders of magnitude worth memorizing: roughly 30 billion microcontrollers ship per year. A mid-range car contains 50–150 electronic control units (Steuergeräte, ECUs) networked over CAN buses. Your washing machine has one microcontroller; your phone contains a dozen besides the main application processor (touch controller, power management, radio coprocessors…). The typical embedded CPU is not a 4 GHz monster — it is a 3.3 V chip running at 16–200 MHz, costing between €0.30 and €5, with everything on one die.

1.1 µC, µP, SoC or FPGA? The decision landscape

  • A microprocessor (µP, Mikroprozessor) is only a CPU. Memory, storage and I/O live in external chips. High performance, high system cost, needs an operating system. (PCs, servers.)
  • A microcontroller (µC, Mikrocontroller) integrates CPU + Flash + SRAM + peripherals (timers, ADC, UART, GPIO…) on one chip. Runs directly from power-on, often with no OS at all. This module's protagonist.
  • A system-on-chip (SoC) is the big sibling: one or more powerful CPU cores plus GPU, radios and memory controllers — a smartphone or Raspberry Pi chip. Runs Linux/Android; boot takes seconds, and timing is no longer deterministic.
  • An FPGA (Module 10/11) is not a processor at all but configurable hardware: thousands of logic blocks you wire up with a hardware description language. Choose it when you need massive parallelism or nanosecond-exact timing that no CPU can guarantee.

2. Inside the CPU — fetch, decode, execute

Strip away marketing and every CPU is a synchronous digital circuit with four essential parts:

  • Registers (Register): a handful of ultra-fast storage cells inside the core. A Cortex-M has 16 × 32-bit registers R0–R15; R13 is the stack pointer (SP), R15 the program counter (PC, Befehlszähler) — the address of the next instruction.
  • The ALU (arithmetic logic unit, Rechenwerk): combinational logic that adds, subtracts, ANDs, shifts — in one clock cycle.
  • The control unit (Steuerwerk): a state machine that reads the current instruction and steers the datapath.
  • Buses to memory: address lines out, data lines in/out.

Execution is a loop that has not changed conceptually since 1945 — the fetch–decode–execute cycle (Befehlszyklus):

\[ \text{fetch: } \mathrm{IR} \leftarrow \mathrm{Mem}[\mathrm{PC}],\quad \mathrm{PC} \leftarrow \mathrm{PC}+4 \;\;\rightarrow\;\; \text{decode} \;\;\rightarrow\;\; \text{execute} \;\;\rightarrow\;\; \text{repeat} \]

2.1 A three-instruction program, cycle by cycle

Here is what the C statement a = a + 5; becomes on a load/store machine (a global a living at SRAM address 0x2000 0004):

LDR  R1, [R0]      ; load:  R1 ← Mem[0x20000004]   (R0 holds the address)
ADD  R1, R1, #5    ; ALU:   R1 ← R1 + 5
STR  R1, [R0]      ; store: Mem[0x20000004] ← R1

Walk it through concretely, assuming the program sits at 0x0800 0100 and a = 7:

  1. Fetch 1: PC = 0x08000100. The control unit puts the PC on the address bus, memory returns the bit pattern of LDR; it lands in the instruction register. PC advances.
  2. Decode/execute 1: the decoder recognizes “load”: put R0’s content (0x20000004) on the address bus, read, write the result 7 into R1. Loads take an extra cycle — memory is slower than registers.
  3. Fetch/execute 2: ADD feeds R1 = 7 and the constant 5 into the ALU; the sum 12 is written back to R1. One cycle.
  4. Fetch/execute 3: STR drives address 0x20000004 and data 12 onto the buses; SRAM stores it. a is now 12.

That is all a CPU ever does — billions of times per second. Every C construct compiles down to sequences like this, which is why a C programmer on a µC can (and sometimes must) reason about individual cycles.

2.2 Pipelining — and its hazards

Fetch, decode and execute use different hardware, so why let two of the three idle? A pipeline (Fließbandverarbeitung) overlaps them: while instruction \(n\) executes, \(n{+}1\) is being decoded and \(n{+}2\) fetched. A Cortex-M3/M4 uses exactly this 3-stage pipeline; big application processors use 10–20 stages. Throughput approaches one instruction per clock even though each instruction still takes three cycles from start to finish.

The catch: hazards (Pipelinekonflikte). If instruction \(n\) is a taken branch, the two instructions already fetched behind it are the wrong ones and must be flushed — a taken branch on Cortex-M costs 2–3 extra cycles. Data hazards (an instruction needing a result the previous one hasn’t written yet) are resolved by forwarding or stalls. This is why “count the instructions” only approximates “count the cycles”, and why hard-real-time engineers read the cycle tables in the core manual.

2.3 Harvard vs. von Neumann, RISC vs. CISC

A von Neumann machine has one memory and one bus for both instructions and data — simple, but fetch and data access compete (the “von Neumann bottleneck”). A Harvard machine has separate instruction and data buses and can do both in the same cycle. Modern µCs are pragmatically hybrid: the Cortex-M is a “modified Harvard” — separate on-chip buses to Flash (instructions) and SRAM (data), but a single unified address space, so you get the speed without the awkwardness.

RISC vs. CISC in one paragraph: CISC (x86) offers hundreds of complex, variable-length instructions — dense code, complicated decoder. RISC (reduced instruction set computer) bets on few, simple, fixed-format instructions executed fast, with all arithmetic register-to-register and memory touched only by load/store. RISC won the embedded world completely: essentially every 32-bit µC today is an ARM Cortex-M (M0+ for cost/power, M4/M7 with DSP and floating point) — ARM licenses the core design to ST, NXP, Nordic, Microchip and everyone else — while the license-free, open-standard RISC-V ISA is the fast-growing challenger, especially in China and in deeply-embedded controller cores inside larger chips (Nvidia GPUs contain dozens). The 8-bit survivors (AVR — the classic Arduino chip — and PIC, 8051) still ship in volume where 50 cents matter more than performance.

A classic beginner accident: logging sensor data to EEPROM every 10 ms wears out its 10⁵-cycle endurance in under 20 minutes.

3.2 The memory map — everything is an address

A 32-bit CPU can address \(2^{32} = 4\,\mathrm{GiB}\). The chip designer carves this space into regions — the memory map (Speicherabbild). Crucially, peripherals live in it too: the UART’s data register, a timer’s counter, the GPIO output latch each have an address, and reading/writing that address is how you talk to hardware. This is memory-mapped I/O (speicherbezogene Ein-/Ausgabe) — no special I/O instructions, just loads and stores.

3.3 Stack, heap, and what happens at reset

Within SRAM, two dynamic regions grow toward each other. The stack (Stapelspeicher) grows downward from the top: every function call pushes its return address and local variables, every return pops them — automatic, fast, and bounded only by how deep your calls nest. The heap (malloc) grows upward. Embedded style guides (MISRA-C, aerospace rules) often ban the heap outright: with 20 kB of RAM, fragmentation or a leak doesn’t slow the system down, it kills it — possibly weeks after deployment. Static allocation is the default culture.

Boot (Startvorgang) is beautifully simple compared to a PC. At release of reset, the Cortex-M hardware reads Flash address 0x00: that word becomes the initial SP. It reads 0x04: that word becomes the PC — the reset handler. The reset handler (a dozen assembly lines generated by your toolchain) copies initialized variables from Flash to SRAM, zeroes the rest (.bss), configures the clock tree, and calls main(). Total time: microseconds to a millisecond. No BIOS, no bootloader, no OS — unless you add one.

🚫Classic pitfall: stack overflow into silence

There is no memory protection by default. If the stack grows past its allotment it silently overwrites your variables (or the heap), and the symptom appears far from the cause — the classic “it crashes every few hours” bug. Defenses: size the stack generously, fill it with a known pattern and check the “high-water mark”, avoid recursion and large local arrays, and enable the MPU or stack-limit registers on cores that have them.

4. GPIO — the pins, deeply

4.1 Port registers and memory-mapped I/O in C

GPIO (general-purpose input/output) pins are organized in ports (GPIOA, GPIOB, …) of 8–16 pins, controlled by a small register set — typically a direction/mode register (is pin 5 an input or an output?), an output data register (ODR: the latch driving the pin), and an input data register (IDR: the live pin state through a Schmitt-trigger input stage). Because of memory-mapped I/O, driving a pin in C is a pointer dereference:

#include <stdint.h>

#define GPIOA_ODR (*(volatile uint32_t *)0x48000014u)  /* output latch  */
#define GPIOA_IDR (*(volatile uint32_t *)0x48000010u)  /* input pins    */

GPIOA_ODR |=  (1u << 5);          /* set PA5 high  -> LED on   */
GPIOA_ODR &= ~(1u << 5);          /* clear PA5     -> LED off  */
GPIOA_ODR ^=  (1u << 5);          /* toggle PA5                */

while (GPIOA_IDR & (1u << 13)) {  /* wait while button (PA13,  */
    /* spin */                     /* active-low) is released   */
}

That volatile qualifier is load-bearing. It tells the compiler: “every read and write of this object is an observable event — never cache it, reorder it away, or merge accesses.” Without it, the optimizer applies ordinary logic: “this loop reads the same variable it never writes, so read it once and decide forever”:

uint32_t *idr = (uint32_t *)0x48000010u;   /* WRONG: volatile missing  */
while (*idr & (1u << 13)) { }              /* at -O2 the compiler reads
                                              once and, if the bit was
                                              set, loops FOREVER        */

The compiler is not wrong — you lied to it. Hardware registers change behind the program’s back, and volatile is how C expresses that. The same applies to variables shared with interrupt handlers (Section 5). Code that “works at -O0 but breaks at -O2” is the classic symptom of a missing volatile.

⚠️Read–modify–write is not atomic

GPIOA_ODR |= (1u << 5); compiles to load–OR–store: three instructions. If an interrupt fires in between and modifies another bit of the same register, its change is overwritten by your stale store. That is why vendors provide atomic set/reset registers (e.g. STM32 BSRR: writing a 1 sets or clears exactly one pin in a single store) — prefer them in interrupt-rich code.

4.2 Push-pull vs. open-drain, and pull-ups

An output pin’s driver stage (a CMOS inverter pair, Module 10) can be configured two ways:

  • Push-pull (Gegentakt): the pin actively drives high (PMOS to \(V_{DD}\)) and low (NMOS to ground). Strong, fast, the default. But two push-pull outputs connected together can fight — one driving high, one low — creating a short circuit.
  • Open-drain (offener Drain): only the pull-down NMOS exists. The pin can pull low or release (high-impedance); an external pull-up resistor (Pull-up-Widerstand) restores the high level. Now many devices may share one wire: the line is high only if everyone releases it — a wired-AND. Any device can assert “low” without ever fighting another driver. This is exactly why I²C (Section 7.3) mandates open-drain, and why open-drain suits shared interrupt/fault lines and level-shifting to a different voltage rail.

Inputs need a defined level too: an unconnected CMOS input floats, picks up 50 Hz hum and switches randomly (and burns crowbar current). Every µC therefore offers internal pull-ups/pull-downs (~20–50 kΩ) you enable per pin — a button to ground plus internal pull-up is the standard two-component input circuit.

4.3 Switch debouncing — Entprellen

Mechanical contacts do not close cleanly: the springy metal bounces (prellt) for 0.1–5 ms, producing a burst of edges. A CPU polling at MHz rates sees every single one — your “one” button press increments the counter by 7.

Two standard cures:

  • RC filter: the capacitor integrates the bounce burst away; the input’s Schmitt trigger (hysteresis, Module 9/10) converts the slow ramp into one clean edge. Dimensioning below.
  • Software: sample the pin on a timer tick (e.g. every 5 ms) and accept a new state only after \(N\) consistent samples — zero extra parts, which is why production firmware almost always debounces in software.
Worked example Dimensioning an RC debounce

A button (bounce spec ≤ 5 ms) pulls the node low; \(V_{CC} = 3.3\) V; the Schmitt input switches high at \(V_{T+} = 1.7\) V. Choose R and C so the input crosses \(V_{T+}\) only ≈ 10 ms after release — twice the worst-case bounce.

Step 1 — charging law (Module 6): after release, C charges through \(R_1 + R_2\): \(v(t) = V_{CC}\,(1 - e^{-t/\tau})\). Solve for the crossing time: \[ t = -\tau \ln\!\left(1 - \frac{V_{T+}}{V_{CC}}\right) = -\tau \ln(1 - 0.515) = 0.724\,\tau . \]
Step 2 — required τ: \( \tau = t / 0.724 = 10\,\mathrm{ms}/0.724 \approx 13.8\,\mathrm{ms} \).
Step 3 — pick parts: with \(R_1 + R_2 = 47\,\mathrm{k\Omega} + 10\,\mathrm{k\Omega} = 57\,\mathrm{k\Omega}\): \(C = 13.8\,\mathrm{ms}/57\,\mathrm{k\Omega} \approx 242\,\mathrm{nF}\) → choose 220 nF (E-series), giving \(t \approx 9.1\) ms. R₂ (10 kΩ) also limits the capacitor’s discharge current into the switch on press.
R₁ = 47 kΩ, R₂ = 10 kΩ, C = 220 nF → clean edge ≈ 9 ms after the last bounce. Sanity check: 9 ms ≫ 5 ms bounce, yet far below the ~50 ms a human can perceive as lag. ✓

5. Interrupts — hardware that taps the CPU on the shoulder

5.1 Polling vs. interrupts

How does the CPU learn that a byte arrived on the UART or a button was pressed? Option one: polling (zyklische Abfrage) — keep asking in a loop. Simple and perfectly deterministic, but the CPU burns cycles asking, and the worst-case reaction time equals your whole loop period. Option two: the interrupt (Unterbrechung) — the peripheral raises a hardware signal, and the CPU suspends whatever it is doing, runs a dedicated function, and resumes as if nothing happened. Reaction time: sub-microsecond, independent of what the main program is doing.

5.2 What actually happens: vectors, context, latency

When an interrupt request (IRQ) arrives and is enabled, the hardware — on a Cortex-M, the NVIC (nested vectored interrupt controller) — performs a precisely choreographed sequence:

  1. Finish the current instruction (you cannot stop mid-instruction).
  2. Save context (Kontextsicherung): push the working registers (R0–R3, R12, LR, PC, status) onto the stack automatically — 8 words.
  3. Vector: fetch the address of the matching interrupt service routine (ISR, Unterbrechungsroutine) from the vector table (Fig. 3.1) and jump.
  4. Run the ISR; on return, pop the context and resume the interrupted code exactly where it stopped.

Steps 1–3 take a fixed 12 clock cycles on a Cortex-M3/M4 — at 80 MHz, that is 150 ns of hardware interrupt latency (Interruptlatenz). Priorities: every IRQ gets a priority level; a higher-priority request preempts a running lower-priority ISR (nesting, geschachtelte Unterbrechungen), while equal/lower ones wait. So the worst-case latency of the highest-priority interrupt stays tiny, and lower ones inherit the sum of everything above them.

Worked example A worst-case latency budget

An 80 MHz Cortex-M4 motor controller must react to an overcurrent comparator within 2 µs. The firmware also has: a UART ISR (measured worst case 4.0 µs), a 1 kHz SysTick ISR (2.5 µs), and short critical sections where all interrupts are masked (longest: 1.2 µs). Overcurrent gets the highest priority. Worst-case reaction?

Step 1 — what can delay the highest priority? Only non-interruptible states: the longest interrupt-masked critical section (1.2 µs) or the tail of one instruction (~ns, negligible) — NOT the other ISRs, because overcurrent preempts them.
Step 2 — add hardware entry: 12 cycles / 80 MHz = 0.15 µs.
Step 3 — total: \( t_{max} = 1.2 + 0.15 \approx 1.35\,\mathrm{µs} + \) first ISR instructions. Budget met with ~30 % margin.
Worst case ≈ 1.4 µs < 2 µs ✓ — and the calculation shows the real lever: keep critical sections short, they gate even your fastest interrupt.

5.3 The golden rules of ISR writing

🔑Key idea: ISRs — short, non-blocking, honest about sharing

1. Keep it short. Every µs inside an ISR is added latency for every equal/lower-priority interrupt. Do the minimum (grab the byte, set a flag, timestamp the edge) and let main() do the thinking.
2. Never block. No delays, no waiting on other hardware, no printf (it may wait on the UART — possibly on an interrupt that cannot run because you are running: deadlock).
3. Declare shared data volatile and mind atomicity: a 64-bit counter or multi-field struct read in main() can be torn mid-read by the ISR updating it. Protect multi-word shared state by briefly disabling that interrupt, or design it away (single-writer flags, ring buffers).

volatile uint32_t ms_ticks = 0;      /* written by ISR, read by main */

void SysTick_Handler(void)           /* runs every 1 ms              */
{
    ms_ticks++;                      /* short: one increment, return */
}

int main(void)
{
    init_hw();
    uint32_t t0 = ms_ticks;
    while ((ms_ticks - t0) < 500) {  /* unsigned wrap-safe 500 ms    */
        /* volatile forces a fresh read each pass                    */
    }
    /* ... */
}

A race condition (Wettlaufsituation) in one line: main() executes count++ (load 41, add, store 42) and the ISR fires between load and store, also incrementing (41→42). The ISR’s increment is lost; the final value is 42, not 43. It will pass every test on your desk and fail once a week in the field. Exercise E5 dissects it.

6. Timers, counters and PWM

6.1 The counter + prescaler arithmetic

A hardware timer (Zeitgeber) is just a counter register incremented by a clock, with helpers: a prescaler (Vorteiler) divides the input clock first; an auto-reload register (ARR, the “period” register) resets the count and can fire an interrupt on overflow. Because both dividers count from zero, each divides by (setting + 1):

\[ \boxed{\; f_{event} = \frac{f_{clk}}{(\mathrm{PSC}+1)\,(\mathrm{ARR}+1)} \;} \]
Worked example An exact 1 kHz tick from a 16 MHz clock

Configure a 16-bit timer (max count 65535) on a 16 MHz µC to interrupt at exactly 1.000 kHz.

Step 1 — total division needed: \( N = f_{clk}/f_{event} = 16\,000\,000 / 1000 = 16\,000 \). Since 16 000 < 65 536, a prescaler of 1 (PSC = 0) would already work with ARR = 15 999.
Step 2 — a nicer factorization: spend the prescaler to get a round tick: PSC = 15 → timer ticks at \(16\,\mathrm{MHz}/16 = 1\,\mathrm{MHz}\), i.e. 1 µs per count. Then ARR = 999 gives 1000 µs per overflow.
Step 3 — verify with the +1 rule: \( f = \frac{16\cdot 10^6}{(15+1)(999+1)} = \frac{16\cdot 10^6}{16\,000} = 1000.0\,\mathrm{Hz} \) ✓ — exact, because 16 MHz is an integer multiple of 1 kHz. Setting PSC = 16 by forgetting the +1 gives 941.2 Hz: a 5.9 % error that would make a software clock lose 85 minutes per day.
PSC = 15, ARR = 999 → exactly 1 kHz, with a convenient 1 µs timer resolution as a bonus.

6.2 PWM — pulse width modulation

Add one compare register (CCR) to the counter and you get PWM (Pulsweitenmodulation): the output pin is high while count < CCR, low for the rest of the period. The duty cycle (Tastgrad) \(D = \mathrm{CCR}/(\mathrm{ARR}{+}1)\) sets the fraction of time spent high, and therefore the average:

\[ \boxed{\; \bar{U} = D \cdot U_{high} \;} \qquad 0 \le D \le 1 \]

Any load that averages — thermally, mechanically, optically or through an LC/RC filter — sees a smoothly adjustable analog value, produced by a purely digital pin at essentially zero loss (the switch is always fully on or fully off; that thermodynamic trick powers all of Module 18). The two canonical uses:

  • LED dimming: at a few hundred Hz and above, your eye integrates the flicker away; perceived brightness follows the duty cycle. Every phone backlight works this way.
  • Hobby servo control: a servo expects a pulse every 20 ms (50 Hz) whose width encodes the position — 1.0 ms = full left, 1.5 ms = center, 2.0 ms = full right. Note the information is in the pulse width (5–10 % duty), not the average.

6.3 Input capture — measuring instead of generating

Timers run in reverse, too: in input capture mode (Messwerterfassung/Capture), an edge on a pin copies the free-running counter value into a capture register and raises an interrupt. Two consecutive rising-edge captures \(c_1, c_2\) give the period without any CPU involvement in the timing itself:

\[ f_{signal} = \frac{f_{timer}}{c_2 - c_1} \]

With the 1 µs timer from the worked example, a fan tachometer reading \(c_2 - c_1 = 8333\) means \(f = 120\) Hz — 60 Hz rotation for a 2-pulse/rev sensor, i.e. 3600 rpm, resolved to ~0.01 %. The same principle measures ultrasonic echo times, RC-receiver pulses and duty cycles (capture both edges).

7. Communication interfaces — teaching chips to talk

A µC rarely lives alone: sensors, memories, displays, radios and other controllers must exchange bytes. Serial buses dominate — fewer pins beat raw speed at these scales. You must know four, and you must know them at the waveform level, because that is what you will stare at on a scope when it doesn’t work.

7.1 UART — the veteran (asynchronous, 2 wires)

The UART (universal asynchronous receiver/transmitter, serielle Schnittstelle) sends each byte as a self-contained frame (Rahmen) with no clock wire at all: TX to RX, one wire per direction. The line idles high (a legacy of current-loop teletypes). A frame is:

  • Start bit (low): its falling edge tells the receiver “a frame begins now” — the only synchronization there is.
  • 8 data bits, LSB first (7/9 possible).
  • Optional parity bit (Paritätsbit): makes the number of 1s even (or odd) — detects any single flipped bit, cheaply.
  • Stop bit (high): guarantees an idle-high gap so the next start edge is detectable.

Both ends must agree on the baud rate (bits per second: 9600 and 115 200 are the classics) and the frame format, because nothing on the wire announces them. The receiver samples each bit in its middle, timed from the start edge by its own local clock; after 10 bits the accumulated clock error must stay below ~half a bit, so the two clocks may differ by at most ≈ 5 % — comfortably met by on-chip oscillators, but the reason exotic baud rates from mismatched crystal frequencies cause garbage. At 115 200 baud 8N1, each byte costs 10 bit times: a hard ceiling of 11 520 bytes/s.

Compose a frame: pick the byte, baud rate and parity, and read the wire like a logic analyzer. Note the LSB goes first — the bit order on the wire is the reverse of how you write the number in binary. Watch the bit time shrink 100× from 9600 baud to 1 Mbaud.

Worked example Decoding a scope capture

A logic analyzer shows an idle-high line that falls, then measures, in units of one bit time (104.2 µs each): 0 · 1 0 1 0 0 1 1 0 · 1. Configuration is 8N1. What was sent, and at what baud rate?

Step 1 — baud rate from bit time: \( 1/104.2\,\mathrm{µs} \approx 9600 \) baud ✓ (the standard rate nearest the measurement).
Step 2 — parse the frame: first 0 = start bit. Next 8 are data, LSB first: D0…D7 = 1,0,1,0,0,1,1,0. Final 1 = stop bit ✓ (a stop-bit violation would flag a framing error).
Step 3 — reassemble MSB…LSB: D7…D0 = 0110 0101 = 0x65 = decimal 101 = ASCII 'e'.
The byte 0x65 ('e') at 9600 baud. Classic trap: reading the data bits left-to-right as written gives 0xA6 — wrong, because UART is LSB-first.

7.2 SPI — the sprinter (synchronous, 4 wires)

SPI (serial peripheral interface) adds what UART lacks: an explicit clock. A master (controller) drives SCK (clock) and MOSI (master-out-slave-in); the slave (peripheral) answers on MISO; a dedicated CS̄ (chip select, active low) line per slave frames the transaction. Data is exchanged simultaneously in both directions — two shift registers forming a ring. No addresses, no acknowledgment, no protocol overhead at all: just raw clocked bits, which is why SPI runs at 10–100 MHz where I²C manages 0.4–3.4 MHz. Cost: it needs \(3 + n\) pins for \(n\) slaves.

Two configuration bits trip everyone once: CPOL (does SCK idle low or high?) and CPHA (is data sampled on the first or second edge?). The four combinations are “modes 0–3”; both ends must use the peripheral’s documented mode. Mode 0 (idle low, sample on rising edge) is the most common:

7.3 I²C — the diplomat (2 wires, many devices)

I²C (inter-integrated circuit, I²C-Bus, Philips 1982) connects up to ~100 devices with just two wires: SDA (data) and SCL (clock), both open-drain with pull-up resistors (Section 4.2) — so any device can pull a line low, none can fight. The protocol earns its pins back with rules:

  • START condition: SDA falls while SCL is high (data normally changes only while SCL is low — the violation is the signal).
  • Addressing (Adressierung): the master sends a 7-bit device address plus a read/write bit. Every chip on the bus compares; only the match responds. (Addresses are baked into the chip, with a few pin-selectable bits — hence the classic conflict of two identical sensors on one bus.)
  • ACK: after every 9th bit slot the receiver pulls SDA low. A missing ACK (NACK) tells the master immediately that nobody is home — priceless for diagnostics.
  • Clock stretching (Taktdehnung): a slow slave may hold SCL low to pause the master — flow control for free, courtesy of open-drain.
  • STOP condition: SDA rises while SCL is high.

7.4 CAN — the diplomat under fire (differential, automotive)

The CAN bus (controller area network, Bosch 1986) is what happens when engineers design a bus for an environment that actively hates electronics: a car. Two measures make it robust. First, the physical layer is differential (CAN_H/CAN_L twisted pair): interference hits both wires equally and cancels in the receiver’s difference (common-mode rejection, Module 9/21). Second — the brilliant part — non-destructive arbitration by ID (Arbitrierung).

Any node may start transmitting when the bus is idle. Collisions are resolved by physics: a transmitted 0 is dominant (drives the bus), a 1 is recessive (releases it — the same wired-AND idea as open-drain). Every transmitter compares the bus with what it just sent, bit by bit, during the message’s identifier field. The moment a node sends recessive 1 but reads back dominant 0, it knows a higher-priority message is on the bus — it stops instantly and retries later. The winner never notices; zero bandwidth is destroyed. Consequence: the numerically lowest ID always wins, so the ID doubles as the message priority — airbag frames get low IDs, seat-heater status gets high ones, and the worst-case latency of the top-priority message is provable. That determinism is why CAN still rules automotive and industrial control 40 years on (typ. 500 kbit/s classic, 2–8 Mbit/s CAN FD).

8. ADC and DAC in a real system

Modules 13 and 15 cover converters in depth; here is the embedded-integration view. The on-chip ADC (typically a 12-bit SAR type, 0.1–5 Msample/s) measures pin voltage against a reference \(V_{ref}\); one code step (LSB) is

\[ 1\,\mathrm{LSB} = \frac{V_{ref}}{2^N} \quad \xrightarrow{\;N=12,\ V_{ref}=3.3\,\mathrm{V}\;} \quad 0.81\,\mathrm{mV} \]

Three things bite in practice. (1) Your measurement is only as good as \(V_{ref}\): if it is just the 3.3 V rail, every ripple on the rail appears in your data — precise designs use a dedicated reference IC. (2) The sample-and-hold needs source impedance low enough to charge its capacitor within the sampling window (datasheets specify e.g. ≤ 10 kΩ); a high-impedance sensor divider needs a buffer op-amp (Module 9) or longer sampling time. (3) When you sample matters as much as how well (Module 12’s sampling theorem): trigger the ADC from a timer, not from software, or your “10 kHz sampling” will jitter with every interrupt — hardware triggering plus DMA (a peripheral that copies results to RAM without the CPU) gives jitter-free acquisition while the CPU sleeps. True DACs are rarer on µCs; PWM + RC filter (Section 6.2) is the poor man’s DAC for slow signals.

9. Real-time — deadlines, not speed

9.1 What “real-time” actually means

Real-time (Echtzeit) does not mean fast. It means: the correctness of the system depends on the time of the result, not only its value — every response must meet its deadline (Zeitschranke). A system that computes the airbag decision perfectly but 50 ms late has failed absolutely (hard real-time); a video decoder that occasionally drops a frame merely degrades (soft real-time). A 16 MHz µC toggling a fuel injector with guaranteed 10 µs precision is a better real-time system than a 3 GHz PC, whose caches, dynamic frequency scaling and OS scheduler make its worst case unknowable. Real-time engineering is the discipline of the worst case, never the average.

9.2 Three software architectures

1. The superloop (Endlosschleife) — the default for small systems:

int main(void)
{
    init_hw();
    for (;;) {                /* the "superloop"                  */
        read_sensors();       /* each step must return quickly!   */
        update_control();
        refresh_display();
        kick_watchdog();      /* prove we are still alive (9.3)   */
    }
}

Simple, inspectable, no OS. Its weakness: worst-case reaction to anything = duration of one whole lap, and one slow step delays everyone.

2. Superloop + interrupts — the workhorse. Time-critical events run in ISRs (Section 5); the loop does the slow thinking on flags and buffers the ISRs fill. This covers the vast majority of shipped products.

3. An RTOS (real-time operating system: FreeRTOS, Zephyr, embOS — kernels of 6–20 kB). It provides tasks (Tasks/Prozesse), each an independent superloop with its own stack, and a scheduler (Ablaufplaner) that always runs the highest-priority ready task, preempting (verdrängend) lower ones the instant a higher one becomes ready — plus queues, semaphores and timers for safe communication. You buy composability (the motor task cannot be delayed by the display task) at the price of new failure modes: race conditions between tasks, deadlocks, and priority inversion — a high-priority task blocked on a mutex held by a low-priority task, which in turn is starved by medium-priority tasks. The standard cure is priority inheritance (the lock holder temporarily inherits the waiter’s priority).

📜Priority inversion on Mars

The 1997 Mars Pathfinder lander famously kept resetting on the surface: a low-priority meteorological task held a shared bus mutex, medium-priority tasks starved it, and the high-priority bus-management task missed its deadline — the watchdog then rebooted the spacecraft. JPL diagnosed it from 200 million km away and fixed it by enabling priority inheritance — a flag in the (VxWorks) RTOS. Every RTOS course tells this story; now you know why.

9.3 The watchdog timer

A watchdog (Watchdog-Zähler) is a free-running hardware down-counter that resets the entire chip when it reaches zero. Correct firmware must “kick” (reload) it periodically — typically every 100 ms–1 s. If the software hangs (infinite loop, corrupted state, latch-up after an ESD hit, a cosmic-ray bit flip), the kicks stop and the watchdog reboots the system into a known-good state within a bounded time. Every serious unattended product ships with the watchdog enabled — it converts “weird firmware death in the field” from a truck roll into a 300 ms hiccup. Two rules: kick it from the main flow only (kicking from a timer ISR is a classic anti-pattern — the ISR happily keeps running while main() is dead), and log the reset cause register so field failures leave evidence.

10. Low power — engineering the sleep schedule

Battery-powered devices live or die by their sleep modes (Schlafmodi). CMOS power is mostly dynamic, \(P \propto C V^2 f\) (Module 10/11), so the recipe is: do the work fast, then stop the clocks and cut the power domains. A typical Cortex-M ladder:

ModeWhat still runsTypical currentWake-up time
Run @ 80 MHzeverything10–30 mA—
Sleepclocks on, CPU halted; any interrupt wakes2–8 mA~1 µs
StopSRAM + registers retained; RTC, some peripherals1–10 µA5–50 µs
Standbyalmost nothing — RTC + backup registers0.03–2 µAms (reboot)

Wake sources (Aufweckquellen) are whatever hardware still breathes in that mode: a GPIO edge (button, sensor alert pin), the real-time clock’s alarm, a UART start edge, a watchdog. The art is duty cycling: sleep at microamps, wake briefly, measure, transmit, sleep again. The battery then sees the time-weighted average current:

\[ \boxed{\; \bar{I} = \frac{I_{act}\,t_{act} + I_{sleep}\,(T - t_{act})}{T} \;} \qquad L \approx \frac{C_{bat}}{\bar{I}} \]
Worked example Battery life of a wireless sensor

A sensor node on a CR2032 coin cell (220 mAh nominal; plan with 200 mAh usable — capacity derates with pulse load and temperature) sleeps at 2 µA and wakes every 60 s for 50 ms, drawing 8 mA while measuring and transmitting. Battery life?

Step 1 — charge per cycle: active: \(8\,\mathrm{mA} \times 0.05\,\mathrm{s} = 0.40\,\mathrm{mA\,s}\); sleeping: \(0.002\,\mathrm{mA} \times 59.95\,\mathrm{s} = 0.12\,\mathrm{mA\,s}\).
Step 2 — average current: \( \bar{I} = (0.40 + 0.12)/60 = 8.7\,\mathrm{µA} \). Note the split: the 50 ms bursts cost 3× more than 59.95 s of sleeping — the burst dominates the budget.
Step 3 — life: \( L = 200\,\mathrm{mAh} / 8.7\,\mathrm{µA} \approx 23\,000\,\mathrm{h} \approx 2.6\,\mathrm{years} \).
≈ 2.6 years — and the worked levers, in order of impact: shorten the radio burst, lengthen the interval, and only then chase sleep current. Halving the 50 ms burst buys more than halving the 2 µA sleep.

11. The development workflow

Embedded development is cross-development: you write and compile on a PC (the host) for a different CPU (the target). The pieces:

  • Cross-compiler (Cross-Compiler), e.g. arm-none-eabi-gcc: produces ARM machine code on your x86 machine. The linker places code/data at the right addresses of Fig. 3.1 using a linker script; output is an .elf/.hex image.
  • Flashing (Flashen): a debug probe (ST-LINK, J-Link) writes the image into Flash via the chip’s SWD pins (serial wire debug, the 2-wire successor of 4-wire JTAG).
  • On-chip debugging: the same SWD link gives the debugger direct hardware access to the running CPU: halt, single-step, read registers and memory, set hardware breakpoints (comparators watching the PC — they work even in Flash) and watchpoints (“halt when anyone writes this address” — the stack-corruption killer).

printf vs. debugger: printf over UART is universal and shows history at a glance, but it is slow (a 50-char line at 115 200 baud blocks ~4.3 ms!), perturbs timing — famously making race conditions vanish while you look (“Heisenbug”) — and is useless in ISRs. The debugger is timing-neutral for inspection but halts the world (bad for a spinning motor). Real projects use both, plus a middle path: log binary events into a RAM ring buffer and read it out over SWD without stopping (ITM/RTT tracing).

HAL vs. bare registers: vendor HALs (hardware abstraction layers, e.g. STM32 HAL) wrap register access in portable C functions — faster to start, portable across the vendor’s families, but bulky and occasionally wrong in edge cases. Register-level code (as in Section 4) is transparent, minimal and exactly as fast as the hardware — but chip-specific. A sane default: HAL for cold paths (init, config), registers for the hot paths (ISRs, bit-banged timing), and always the reference manual open, because the HAL is somebody’s interpretation of it.

12. Embedded C — the dialect

12.1 Fixed-width types and bit manipulation

Plain int is 16 bits on an AVR and 32 on ARM — porting code that assumed one or the other is a rite of passage you can skip: embedded C uses the exact-width types of <stdint.h> (uint8_t, int16_t, uint32_t…) whenever size matters, i.e. for registers, protocols and arrays. Hardware speaks in bits, so the four mask idioms below are the vocabulary you will type daily. With x = 0110 1010₂ and \(n = 2\):

GoalIdiomResult on x = 0110 1010Why it works
set bit nx |= (1u << n);0110 1110OR with 1 forces 1, OR with 0 keeps

⚠️The 1u is not pedantry

1 << 31 shifts a signed int into its sign bit — undefined behavior in C. 1u << 31 is well-defined. On 16-bit-int machines, 1u << 20 is still wrong (1ul needed). Compilers may exploit undefined behavior to “optimize” your code into surprises; MISRA-C exists for reasons like this.

12.2 Fixed-point arithmetic — Festkommaarithmetik

Many small cores have no floating-point hardware; a software float multiply costs ~100 cycles. The embedded alternative: fixed-point — store \(x \cdot 2^{15}\) as an integer (“Q15” format: 1 sign bit, 15 fraction bits, range −1…+0.99997, resolution \(2^{-15} \approx 3\cdot10^{-5}\)). Then addition is plain integer addition, and multiplication needs one shift to renormalize:

typedef int16_t q15_t;                    /* value = raw / 32768      */

static inline q15_t q15_mul(q15_t a, q15_t b)
{
    return (q15_t)(((int32_t)a * b) >> 15);   /* 32-bit product, then
                                                  drop 15 fraction bits */
}
/* 0.5 * 0.25: a = 16384, b = 8192 -> (134217728 >> 15) = 4096 = 0.125 */

One integer multiply plus a shift: single-digit cycles. Your DSP filters (Module 13) and motor-control loops (Module 14/17) run in Q15/Q31 on small cores exactly this way; the price is that you manage the scaling and watch for overflow at every step.

14. Exercises

Reach for a datasheet mindset: exact registers, exact bits, exact microseconds.

E1A control register holds x = 0xB4. In C: (a) set bit 0, (b) clear bit 7, (c) toggle bit 2, (d) test whether bit 5 is set. Give each idiom and the resulting value of x after applying (a)–(c) in sequence. easy

Show solution
0xB4 = 1011 0100. (a) x |= (1u << 0); → 1011 0101 = 0xB5. (b) x &= ~(1u << 7); → 0011 0101 = 0x35. (c) x ^= (1u << 2); → 0011 0001 = 0x31. (d) if (x & (1u << 5)) … — on 0x31 = 0011 0001 bit 5 is 1 → true. Sanity check: each operation touched exactly one bit; the other seven never changed. That single-bit discipline is the entire point of the mask idioms.

E2An LED on a 3.3 V pin is PWM-dimmed at 1 kHz with CCR = 300, ARR = 999. Find the duty cycle, the average pin voltage, and the average LED current if the LED+resistor draws 12 mA when the pin is high. Why does 1 kHz look flicker-free? easy

Show solution
\( D = 300/1000 = 30\,\% \); \( \bar U = 0.3 \cdot 3.3 = 0.99\,\mathrm{V} \); \( \bar I = 0.3 \cdot 12 = 3.6\,\mathrm{mA} \). The eye's flicker-fusion threshold is ≈ 60–90 Hz (higher in peripheral vision); at 1 kHz each period is 1 ms and the retina integrates perfectly — you perceive only the 30 % average. Camera rolling shutters are less forgiving, which is why cheap PWM lighting shows banding in videos.

E3Configure a 16-bit timer on a 16 MHz clock to drive a hobby servo: 50 Hz PWM with 1 µs pulse-width resolution. Give PSC and ARR, and the CCR values for full left (1.0 ms), center (1.5 ms) and full right (2.0 ms). Would PSC = 0 also work? medium

Show solution
1 µs per count needs the counter clocked at 1 MHz: \( \mathrm{PSC} + 1 = 16 \Rightarrow \mathrm{PSC} = 15 \). Period 20 ms = 20 000 counts → \( \mathrm{ARR} = 19\,999 \). Check: \( f = 16\,\mathrm{MHz}/(16 \cdot 20\,000) = 50\,\mathrm{Hz} \) ✓. CCR = 1000 (left), 1500 (center), 2000 (right) — pulse width in µs directly, a pleasant side effect of the round prescaler. PSC = 0 would need ARR = 319 999 > 65 535: does not fit in 16 bits. The prescaler exists precisely to trade resolution for range.

E4A logger streams 20-byte records over UART at 115 200 baud, 8E1 (8 data, even parity, 1 stop). How long does one record take, what is the maximum record rate, and what fraction of the raw bit rate is payload? medium

Show solution
8E1 frame = 1 start + 8 data + 1 parity + 1 stop = 11 bits → \( t_{byte} = 11/115\,200 = 95.5\,\mathrm{µs} \). Record: \( 20 \cdot 95.5\,\mathrm{µs} = 1.91\,\mathrm{ms} \) → max ≈ 523 records/s. Payload efficiency: 8 of 11 bits = 72.7 %. So the "115 200 baud" link moves at most \(115\,200 \cdot 8/11 \approx 83.8\,\mathrm{kbit/s}\) of data. Real-world remark: if the firmware writes records with a blocking printf, the CPU is trapped for those 1.9 ms per record — at 500 records/s that is ~100 % CPU. This is why logging uses interrupt- or DMA-driven TX buffers.

E5Main code runs events++; on a volatile uint32_t while a UART ISR also runs events++;. The compiler translates each increment into load–add–store. Construct the exact interleaving that loses an update, state how many increments can be lost per collision, and give two independent fixes. medium

Show solution
Interleaving with events = 41: main loads 41 → ISR fires: loads 41, adds, stores 42, returns → main adds (41+1) and stores 42. Two physical increments, counter advanced by one: exactly one update lost per such collision (each collision window loses at most the ISR's one increment). Note volatile did not help — it forces real loads/stores but cannot glue three instructions into one atomic operation; that is the classic misconception. Fixes: (1) make the read-modify-write atomic — briefly disable the UART interrupt around events++ in main (or use a hardware atomic like LDREX/STREX or <stdatomic.h>); (2) redesign to single-writer: only the ISR increments; main keeps its own private count of processed events and compares — no shared read-modify-write exists anymore. Single-writer designs are the embedded gold standard.

E6A tracker on a 200 mAh (usable) cell sleeps at 1.5 µA; each wake-up burst costs 10 mA for 20 ms. Marketing demands 5 years of life. Find the maximum allowed average current, the largest share available for bursts, and the minimum wake interval. What interval would double the remaining margin actually require? challenge

Show solution
5 years = \(5 \cdot 8766 \approx 43\,830\,\mathrm{h}\) → \( \bar I_{max} = 200\,\mathrm{mAh}/43\,830\,\mathrm{h} = 4.56\,\mathrm{µA} \). Sleep already uses 1.5 µA → burst budget \( \bar I_b = 3.06\,\mathrm{µA} \). One burst costs \( Q = 10\,\mathrm{mA} \cdot 0.02\,\mathrm{s} = 0.2\,\mathrm{mA\,s} = 200\,\mathrm{µA\,s} \); average from bursts every \(T\): \( \bar I_b = Q/T \). \( T_{min} = 200\,\mathrm{µA\,s}/3.06\,\mathrm{µA} \approx 65\,\mathrm{s} \) → wake at most every ~65 s (e.g. once a minute is just barely too often: 200/60 = 3.33 µA > 3.06 µA — the calculation catches it!). Doubling margin means halving \( \bar I_b \) to 1.53 µA → \( T = 131\,\mathrm{s} \). Sanity check: the sleep floor alone (1.5 µA) already consumes \(1.5/4.56 = 33\,\%\) of the total budget — at 5-year scale, sleep current is no longer negligible.

E7Three CAN nodes start transmitting simultaneously with 11-bit identifiers 0x2A7, 0x29F and 0x2B1. Write out the arbitration bit by bit: who wins, at which bit does each loser drop out, and in what order do all three messages ultimately cross the bus? challenge

Show solution
MSB-first, 11 bits: 0x29F = 010 1001 1111, 0x2A7 = 010 1010 0111, 0x2B1 = 010 1011 0001. Bits 10…6 (01010) are identical — all three continue, bus = each bit (dominant where any sends 0). Bit 5: 0x29F sends 0 (dominant), 0x2A7 and 0x2B1 send 1 (recessive) but read the bus low → both drop out at bit 5. 0x29F completes: lowest ID wins, as it must. Retry round: 0x2A7 vs 0x2B1 agree on 010101 (bits 10–5); bit 4: 0x2A7 sends 0, 0x2B1 sends 1 → 0x2B1 drops out at bit 4. Order on the bus: 0x29F, then 0x2A7, then 0x2B1 — exactly ascending IDs. Remark: no bit was ever corrupted and no backoff-and-retry randomness was needed (compare Ethernet's collision handling) — arbitration cost zero bus time. That is why the ID is the priority and why safety messages get low IDs at design time.
